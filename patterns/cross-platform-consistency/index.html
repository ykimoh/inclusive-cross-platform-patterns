Pattern 12 · Cross-cutting

Cross-platform consistency

How a single accessibility pattern maps consistently across web, iOS, and Android. Behavior parity first; visual style can diverge by necessity.

Problem

Cross-platform consistency is typically mapped at the visual level — same brand color, same font, same spacing — but inconsistency at the accessibility level is where users actually pay the cost. A focus indicator that’s a 2px ring on web, a system halo on iOS, and a Material elevation on Android: the user who navigates by keyboard or switch control loses cross-platform muscle memory at every surface boundary.

The problem is sharper than aesthetics. The same product can meet WCAG 2.1 on web and ship a custom-rolled iOS modal that traps focus, because the platform-mapping work was treated as visual rather than structural. Accessibility primitives differ across platforms; consistency at the accessibility level is a separate, deliberate exercise.

Pattern

For each accessibility-relevant component, document three platform mappings explicitly. The visual style can differ; the focus order, label semantics, and announcement order must not.

Pattern Web iOS Android
Focus indicator :focus-visible ring ≥3:1 System accessibility focus halo View focus highlight
Form label <label for> + aria-describedby accessibilityLabel, accessibilityHint contentDescription, labelFor
Landmark / region <main>, <nav>, <header> .accessibilityElement + .header trait AccessibilityRole.HEADER
Heading order <h1>–<h6> accessibilityTraits = .header setAccessibilityHeading(true)
Image alt text alt, alt="" for decorative accessibilityLabel, isAccessibilityElement contentDescription, null for decorative

Three rules:

  1. Behavior parity first. Visual style can differ across platforms; focus order, label semantics, and announcement order should not.
  2. Use platform-native primitives whenever they exist. A custom-rolled focus ring on iOS is worse than the system halo for users who have system-level accessibility customizations applied; the same goes for TalkBack on Android.
  3. Document the divergence. When web uses a custom modal and iOS uses the native sheet, the team should know what’s different and why — and verify the accessibility behavior on each.

Reference

Young Kim’s 2023 MFA thesis (Beyond) is a working example of cross-platform accessibility work at the tool level: the prototype ships as an integrated Figma plugin plus a Desktop app, applying the same field-level patterns — color contrast, UX copy, gender identity form library, keyboard navigation, structure landmarks, alt text — to designs that themselves ship across web, iOS, and Android. The split is by where the designer works (audits where they design = Figma; annotations where they document for handoff = standalone), not by where the design ships. The pattern stays stable; the surface adapts.

How to use

  • Define mappings early. Build the platform-mapping table during design-system definition, before component implementations diverge. Retrofitting is harder than starting with the table.
  • Spec by primitive, not by component. In every component spec, name the three implementations and the chosen accessibility primitive for each. Don’t leave it to per-platform engineering judgment.
  • Ask the parity question in review. “Does this focus indicator / label / landmark match the spec across all three platforms?” If the answer is “we’ll check later,” it won’t happen.

What this is not

  • Not a mandate for visual sameness. The brand can still feel native on each platform; the parity is at the accessibility-behavior layer.
  • Not a substitute for per-platform testing (#10). A mapping table is necessary but not sufficient — VoiceOver and TalkBack still need to be run on real builds.
  • Not a one-time exercise. Platform accessibility APIs evolve; revisit the table when the platforms ship significant accessibility updates.

WCAG mapping

WCAG applies on web; iOS and Android map to platform-native equivalents that satisfy the same intent. The mapping table is the bridge.

  • 4.1.2 Name, Role, Value (opens in new tab) A applies on all three platforms; primitives differ
  • 2.4.7 Focus Visible (opens in new tab) AA visual style differs across platforms; visibility requirement is the same
  • 2.4.6 Headings and Labels (opens in new tab) AA same intent across platforms; expressed via different primitives
